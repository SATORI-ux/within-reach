# UX Guidelines

## UX goal
The interface should feel like a quiet shared object, not a software system asking to be operated.

Restraint is part of the user experience, not a decorative choice.

---

## Emotional design targets
The product should read as:

- warm
- quiet
- personal
- restrained
- softly alive
- curated rather than chaotic

It should not read as:

- feature-rich
- gamified
- highly interactive for its own sake
- socially performative
- analytically instrumented
- visually loud

---

## Page composition
Use one long soft scroll.

Preferred order:
1. landing / arrival
2. quick actions
3. check-ins
4. notes
5. quiet ending

This order is stable and should not be rearranged casually.

---

## Arrival guidelines
Arrival should feel like:
- a breath
- a recognition moment
- a threshold into the shared space

It should not feel like:
- a loading screen
- a splash screen
- a gimmick
- a theatrical entrance

### Arrival rules
- Resolve identity before showing personalized copy.
- Avoid generic greeting flashes before visitor truth is known.
- Motion should be subtle and brief.
- Skip paths should exist through click, tap, or scroll.
- Ambient copy may vary, but labels and navigation should remain stable.
- Secondary copy should remain visually subordinate.

---

## Quick action guidelines
The quick actions are the product’s center of gravity.

### Thinking of you
- should feel easiest
- should feel ambient rather than transactional
- should confirm gently
- should not create visible scorekeeping

### Leave a note
- should open into a calm, simple composition flow
- should feel more reflective than reactive
- should not resemble a chat composer

### Can you talk?
- should feel visually distinct
- should not use panic styling
- should require calm confirmation
- should resolve privately

Do not add extra action categories without a strong product reason.

---

## Check-ins vs notes
These surfaces must remain visibly and conceptually separate.

### Check-ins
Should feel:
- lighter
- shorter
- more ambient
- less visually heavy

Do not add:
- reactions
- large cards
- dense metadata
- chat-like styling

### Notes
Should feel:
- readable
- chronological
- held
- calmer than messaging

Notes should show:
- sender
- timestamp
- content
- tiny curated reactions

Reactions should remain limited, quiet, and secondary.

---

## Copy guidelines
Product copy should be:
- warm
- restrained
- direct
- lightly literary
- calm

Avoid:
- startup phrasing
- feature explanation copy
- over-earnest sentimentality
- jokiness that breaks tone
- system language exposed to the user unless necessary

### Randomness rules
Controlled randomness is allowed only where it helps the product feel alive.

Good uses:
- arrival lines
- funny facts or secondary lines
- check-in phrase variants

Bad uses:
- labels
- urgent flows
- navigation
- critical state explanations

Randomness should never make the app feel inconsistent.

---

## Visual guidelines
Favor:
- warm neutrals
- muted accents
- whitespace
- soft section framing
- literary typography
- understated shadows
- subtle motion
- visual stillness after reveal

Avoid:
- bright app colors
- dashboard surfaces
- stat blocks
- cluttered controls
- large icon systems
- aggressive banners
- heavy modal energy

The product should look handcrafted, not optimized.

---

## Feedback behavior
Feedback should be ephemeral and proportional.

Use:
- fading confirmation toasts
- accent-colored but quiet success states
- small inline confirmations when needed

Avoid:
- persistent counters
- large celebratory states
- repeated state labels
- feedback that becomes part of the page chrome

---

## Footer / ending guidelines
The footer matters more than it appears to.

It should feel:
- understated
- settled
- low-affordance
- emotionally complete

Do not turn the footer into:
- a utility area
- a settings zone
- a control bar
- an obvious interactive affordance

Even when it holds hidden mechanics in private builds, it must still read as footer copy first.

---

## Feed growth guidance
Unbounded feed growth weakens the object-like quality of the app.

Preferred pattern:
- recent items only
- quiet “show older” reveal
- progressive access to deeper history

Avoid:
- full endless expansion
- loud pagination UI
- archive-heavy framing

The page should still feel small after repeated use.

---

## Review checklist for UX changes
Before accepting a UX change, ask:

- Does this make the page feel more like a productized app?
- Does this increase cognitive load?
- Does this weaken the distinction between the three actions?
- Does this add visible complexity for little emotional value?
- Does this harm the quiet ending?
- Does this make urgency feel louder than intended?
- Does this reduce warmth or readability?
- Does this make the interface less small?
